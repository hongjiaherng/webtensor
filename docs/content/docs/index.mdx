**webtensor** is a multi-backend tensor library built for the browser. Write, train, and visualize models in TypeScript with three backends at full parity: **CPU** (TypeScript), **WASM** (Rust via wasm-pack), and **WebGPU** (WGSL compute shaders). 765 cross-backend tests.

> **Status: foundation done.** Broadcast-aware autograd, PyTorch-style `matmul` (incl. N-D batched), reductions, softmax, JAX-style `compile()` + `grad()`, host-side `SGD`, end-to-end XOR MLP trains on all three backends. See [Start here](/docs/onboarding/start-here) for the full status matrix.

---

## Two layers: high-level and low-level API

Every user action can be done at two levels. The **high-level** API is what 99% of users want. The **low-level** primitives are there for custom compilers, ONNX round-tripping, and library internals.

### High-level (idiomatic)

```ts
import { tensor, randn, add, matmul, compile, grad, run } from '@webtensor/core';
import { relu, mseLoss } from '@webtensor/nn';
import { SGD } from '@webtensor/nn';
import '@webtensor/backend-webgpu'; // pick one (or several)

// Eager one-shot — `run()` evaluates the tensor graph:
const y = await run(add(tensor([1, 2, 3]), tensor([4, 5, 6])));
console.log(y.data); // Float32Array [5, 7, 9]

// Or chain via the method form:
const z = await tensor([-1, 2, -3]).abs().run();

// Shape-aware equality:
expect(y.equals(tensor([5, 7, 9]))).toBe(true);

// Training — JAX-style: trace once, call many
const W = randn([2, 8], { requiresGrad: true, std: 0.5 });
const b = randn([8], { requiresGrad: true });

const step = await compile(
  ({ x, y }) => {
    const pred = relu(add(matmul(x, W), b));
    const loss = mseLoss(pred, y);
    return { loss, dW: grad(loss, W), db: grad(loss, b) };
  },
  { x: [4, 2], y: [4, 8] },
  { device: 'webgpu' },
);

const opt = new SGD(0.1);
for (let i = 0; i < 1000; i++) {
  const { loss, dW, db } = await step({ x: xBatch, y: yBatch });
  opt.step([W, b], [dW, db]);
}
```

### Low-level (when you need it)

```ts
import { tensor, add, compileGraph, Engine, placeholder } from '@webtensor/core';
import { CPUBackend } from '@webtensor/backend-cpu';

// Explicit graph → engine → read-back: what `run()` does internally.
const y = add(tensor([1, 2, 3]), tensor([4, 5, 6]));
const graph = compileGraph([y]);
const engine = new Engine(await CPUBackend.create());
await engine.evaluate(graph);
const data = await engine.get(y.id); // Float32Array

// Placeholder + feeds: what `compile()` wires up for you.
const x = placeholder([4, 2]);
const W = placeholder([2, 8]);
W.requiresGrad = true;
const out = matmul(x, W);
const graph2 = compileGraph([out]);
await engine.evaluate(graph2, { [x.id]: xData, [W.id]: wData });
```

The low-level API is stable and documented. It's what an ONNX importer or a custom JIT would use. Stick with the high-level API unless you're extending the library.

## Packages

![Package dependencies](/diagrams/package-dependencies.svg)

- `@webtensor/ir` — pure data: `Node`, `Value`, `Graph`, `DType`.
- `@webtensor/runtime` — `Engine`, `Backend` interface, view ops, dtype utilities.
- `@webtensor/core` — user-facing tensor library: `Tensor`, ops, autograd, `compile` / `grad` / `run`.
- **`@webtensor/nn`** — activations (`relu`, `sigmoid`, `tanh`, `softmax`) and losses (`mseLoss`, ...).
- **`@webtensor/nn`** — host-side optimizers (`SGD`; Adam coming soon).
- `@webtensor/backend-cpu` / `-wasm` / `-webgpu` — kernel implementations.

Packages split like **JAX + optax + flax** and **tfjs-core + tfjs-layers** — ML-specific building blocks (`nn`, `optim`) sit on top of the tensor core, not inside it. Inference-only users don't pay the training-code bundle cost.
