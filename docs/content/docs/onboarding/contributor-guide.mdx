## Local dev loop

### Prerequisites

- [Bun](https://bun.sh) (the package manager and runner).
- [`wasm-pack`](https://rustwasm.github.io/wasm-pack/) (only if you touch the WASM backend).
- [PlantUML](https://plantuml.com/) on `PATH` (only if you regenerate diagrams).

### First-time setup

```sh
bun run setup            # install workspace + docs deps
bun run build            # build all 8 packages (chains wasm-pack → bun build → tsc for backend-wasm)
bun run test             # CPU + WASM + WebGPU parity (1e-5 tolerance)
```

`bun run build` rebuilds every package in dependency order. The `backend-wasm` build chains three steps under the hood: `wasm-pack build` writes `packages/backend-wasm/pkg/` (git-ignored), `bun build.ts` bundles `src/` and inlines the `.wasm` binary as base64 into `dist/index.js`, and `tsc --emitDeclarationOnly` writes the `.d.ts` files. If you're only iterating on the WASM kernels, `bun run build:backend-wasm` rebuilds just that package.

### Day-to-day commands

| Command                              | What it does                                                                                                                      |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| `bun run test`                       | Run the full Vitest suite in headed Chromium. **Do not use `bun test`** — that bypasses the browser config and WebGPU tests fail. |
| `bun run test:watch`                 | Vitest in watch mode.                                                                                                             |
| `bun run test tests/ops/binary/add.test.ts` | Run one test file.                                                                                                         |
| `bun run lint`                       | ESLint (currently has 1 known error — see [Bugs and gaps](/docs/onboarding/bugs-and-gaps)).                                       |
| `bun run lint:fix`                   | ESLint with auto-fix.                                                                                                             |
| `bun run format`                     | Prettier write.                                                                                                                   |
| `bun run format:check`               | Prettier check (CI-style).                                                                                                        |
| `bun run build`                      | Build all eight packages to `dist/`. Sequential — runs `ir → runtime → core → backend-cpu → backend-webgpu → backend-wasm → nn → optim`. |
| `bun run clean`                      | Delete `dist/` outputs.                                                                                                           |
| `bun run clean:all`                  | Also delete `node_modules` and the `pkg/` WASM bundle.                                                                            |
| `bun run docs:plantuml`              | Render `docs/diagrams/puml/*.puml` to `*.svg` into `docs/public/diagrams/` via PlantUML.                                          |
| `bun run docs:dev`                   | Start the fumadocs (Next.js) dev server for the docs site.                                                                        |
| `bun run docs:build`                 | Build the docs site as a static export (for GitHub Pages).                                                                        |

## Adding a new op

The canonical procedure lives in [Adding an op](/docs/advanced/adding-an-op). High-level it's four steps × three backends:

1. **Core graph function** in [packages/core/src/ops/&lt;category&gt;/&lt;op&gt;.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/core/src/ops) — declares the op name, output shape, attributes, and backward closure.
2. **CPU kernel** in `packages/backend-cpu/src/kernels/<category>/<op>.ts`, registered in `kernels/registry.ts`.
3. **WASM kernel** — Rust function in `packages/backend-wasm/rust/src/ops/<category>/<op>.rs`, JS wrapper in `packages/backend-wasm/src/kernels/<category>/<op>.ts`, register, **rebuild WASM** with `wasm-pack`.
4. **WebGPU kernel** — `<op>.wgsl` (uniform must be `u_meta` not `meta`!) plus `<op>.ts` implementing `WebGPUKernel`, register.

Then:

5. **Per-op test** in `tests/ops/<category>/<op>.test.ts`.
6. **Cross-backend parity** — make sure `tests/backend/consistency.test.ts` exercises the new op on all three backends.

Add the op to [Backends and kernels](/docs/onboarding/backends-and-kernels) parity table and update [Roadmap](/docs/roadmap) if it was on the checklist.

## Test patterns

### Use `bun run test` not `bun test`

This is the single most common new-contributor trip-up. Vitest browser config is loaded by `vitest`, not by Bun's built-in test runner. Using `bun test` runs in Node and the WebGPU tests fail with `navigator.gpu is undefined`.

### Build graphs in `beforeAll`, not in `describe` body

`Tensor.id` comes from a process-global counter ([tensor.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/core/src/tensor.ts)). Vitest can run multiple test files in the same process. If you build graphs at module load (the `describe` body), tensor IDs from earlier files collide with later ones and tests interfere.

```ts
describe('add', () => {
  let graph: Graph;
  beforeAll(() => {
    const a = tensor([1, 2, 3]);
    const b = tensor([4, 5, 6]);
    graph = compileGraph([add(a, b)]);
  });
  // ...
});
```

### Use the helpers in `tests/helpers.ts`

[`tests/helpers.ts`](https://github.com/hongjiaherng/webtensor/blob/main/tests/helpers.ts) exports:

- `BACKENDS` — array of `{ name, create }` entries to iterate over for parity tests. Each `create()` returns a fresh `Backend` instance.
- `DTYPES` — dtype axis (`['float32', 'int32']`) for tests that need dtype × backend coverage.
- `expectClose(actual, expected, tol = 1e-5)` — float comparison.

Tolerance is `1e-5`. CPU is the correctness oracle — WASM and WebGPU are compared against CPU output, not against hand-computed expected values.

### Test layout

| Folder              | What lives there                                      |
| ------------------- | ----------------------------------------------------- |
| `tests/ops/<category>/` | One file per op, grouped by category (`binary/`, `unary/`, `activation/`, `reduce/`, `compare/`, `linalg/`, `view/`, `shape/`, `cast/`, `init/`). |
| `tests/backend/`    | Cross-backend `consistency.test.ts`, GC/memory tests. |
| `tests/foundation/` | Dtype round-trip, view ops, engine internals.         |
| `tests/graph/`      | Autograd, `compile()`, placeholder, training loop.    |

## Common pitfalls

### WGSL `meta` is reserved

Always name your uniform `u_meta` (or `u_meta_a` / `u_meta_b`). Using bare `meta` produces a silent compile failure — the pipeline builds and every output is zero. Tests pass `expectClose(0, 0)` if you write your assertions wrong, so this can hide.

### Bind-group binding indices must be consecutive

WebGPU rejects bind groups with gaps. Unary kernels use 0,1,2; binary kernels use 0,1,2,3,4. Don't skip.

### WebGPU temp buffers must outlive submission

In [packages/backend-webgpu/src/backend.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/src/backend.ts), `tempBuffers` are destroyed only after `device.queue.onSubmittedWorkDone()` resolves. If you destroy earlier the GPU reads garbage.

### `Backend.dispose()` is a no-op for views

When `tensor.isView === true`, `dispose` must do nothing — the parent owns the storage. This is enforced in every backend; don't accidentally `buffer.destroy()` a view's storage.

### Engine's `evaluate` is `async`

Because WebGPU returns a Promise from `execute()`. Always `await engine.evaluate(graph)` — losing the await on WebGPU means you read `undefined` from `engine.get()` because the kernel hasn't run.

### Reshape vs View

`reshape()` auto-copies if the source is non-contiguous (PyTorch semantics). `view()` is strict — throws if non-contiguous. When in doubt, use `reshape()`.

### Rebuild WASM after touching Rust

`wasm-pack build` does not run automatically. After any change to `packages/backend-wasm/rust/`, re-run the build command from the top of this file.

### Test-tensor-id discipline

Tests that share a `describe` block but build their own graphs at top-level will see ID collisions across runs. Always wrap construction in `beforeAll` or `it.each` — don't build at module evaluation.

## Submitting a change

There is no formal CONTRIBUTING.md or PR template. The de-facto checklist:

1. `bun run test` — green across all three backends.
2. `bun run lint` and `bun run format:check` — green (modulo the known ESLint error on `scripts/clean.mjs` from [Bugs and gaps](/docs/onboarding/bugs-and-gaps)).
3. If you added an op: update the parity table in [Backends and kernels](/docs/onboarding/backends-and-kernels) and the roadmap checklist in [Roadmap](/docs/roadmap).
4. If you fixed a documented bug: remove it from [Bugs and gaps](/docs/onboarding/bugs-and-gaps).

Continue with [Rearchitect notes](/docs/history/rearchitect-notes) if you're planning bigger work.
