Three backends, one contract, full parity. This file goes deep on each.

For the canonical recipe to add a new op see [Adding an op](/docs/advanced/adding-an-op). This file is the **why and how it fits together** â€” the patterns, the gotchas, the specific layouts.

---

## Common pattern â€” registry + factory

Every backend follows the same shape:

1. A **kernel registry** â€” `Map<string, KernelFn>`, op name â†’ kernel.
2. A **`Backend` class** implementing `allocate / read / write / execute / dispose`.
3. An **`index.ts`** that calls `registerBackend(device, factory)` at module load.
4. The factory is `() => Promise<Backend>` â€” async because WASM and WebGPU need init.

`Engine.create(device)` looks up the factory in `backendRegistry` ([engine.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/runtime/src/engine.ts)), awaits it, and wraps the resulting `Backend` in an `Engine`.

`backend.execute(node, inputs, outputs)` is one line: look up `node.op` in the registry, call the kernel. If missing, throw `unsupported op`.

## CPU backend

[packages/backend-cpu](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-cpu/) â€” the simplest and the correctness oracle.

### Backend class

[backend.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-cpu/src/backend.ts):

- `allocate()` â€” `new TypedArray(size)` via `typedArrayCtor(dtype)` from `@webtensor/runtime/dtype`.
- `read()` â€” returns the typed array directly.
- `write()` â€” `tensor.storage.buffer.set(data)`.
- `execute()` â€” registry lookup + call.
- `dispose()` â€” sets `tensor.storage.buffer = null` (no-op for views).

### Kernel anatomy

`CPUKernel = (node, inputs, outputs) => void`. Every kernel handles arbitrary strides via `stridedIdx()`.

Example â€” [`addKernel`](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-cpu/src/kernels/elementwise/add.ts):

```ts
const aBcast = broadcastStridesOf(outShape, aShape, inputs[0].strides);
const bBcast = broadcastStridesOf(outShape, bShape, inputs[1].strides);
for (let i = 0; i < outBuf.length; i++) {
  outBuf[i] =
    aBuf[stridedIdx(outShape, aBcast, inputs[0].offset, i)] +
    bBuf[stridedIdx(outShape, bBcast, inputs[1].offset, i)];
}
```

The pattern is identical for `sub`, `mul`, `div` â€” only the operator changes.

Unary kernels (`relu`, `exp`, `tanh`, â€¦) are even simpler: one `stridedIdx` per element.

`MatMul` ([linalg/matmul.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-cpu/src/kernels/linalg/matmul.ts)) loops over flat batch indices (using `broadcastStridesOf` to handle broadcast-aligned batch strides of A and B), then runs a manual `M Ă— N Ă— K` triple loop per batch using row/col strides directly.

`Contiguous` ([memory/contiguous.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-cpu/src/kernels/memory/contiguous.ts)) is a pure copy from strided source â†’ contiguous destination via `stridedIdx`.

### Performance profile

Single-threaded. Slow for large tensors but always works â€” useful as a fallback and for correctness checks.

## WASM backend

[packages/backend-wasm](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/) â€” Rust kernels via `wasm-bindgen`, called from TypeScript with raw pointers.

### Build

```sh
bun run build:backend-wasm
```

This chains three steps: `wasm-pack build rust --target bundler --out-dir ../pkg` (compiles Rust to WASM), then `bun build.ts` (Bun bundler with a custom plugin that inlines the `.wasm` binary as a base64 string), then `tsc --emitDeclarationOnly` (emits `.d.ts` files). Re-run whenever Rust source changes. Only `dist/` is published; `pkg/` is build-time only.

### Module loading (base64 inline)

The published `dist/index.js` contains the `.wasm` binary inline as a base64 string â€” consumers don't need `vite-plugin-wasm` or any other bundler config. [src/module.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/src/module.ts) decodes the base64 and instantiates the module manually:

```ts
import wasmBase64Raw from '../pkg/webtensor_wasm_bg.wasm'; // bundler â†’ base64 string
const wasmBase64 = wasmBase64Raw as unknown as string;

const bytes = Uint8Array.from(atob(wasmBase64), (c) => c.charCodeAt(0));
const { instance } = await WebAssembly.instantiate(bytes, {
  './webtensor_wasm_bg.js': {
    __wbindgen_init_externref_table: () => {}, // no-op: kernels use only numeric pointers
  },
});
instance.exports.__wbindgen_start();
```

`loadWasmModule()` is genuinely async and caches the result â€” the WASM is instantiated lazily on the first `WASMBackend.create()` call, not at import time.

### Tensor handle

`storage.buffer` is a `WasmTensorHandle = { ptr, elements, byteLength }` â€” a pointer into WASM linear memory plus its length. The kernel passes `ptr` to Rust; Rust dereferences it via `unsafe`.

### Memory management

Rust ([rust/src/memory.rs](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/rust/src/memory.rs)) exposes `alloc_f32` / `free_f32` (and the `u32` pair) using `Vec::with_capacity` + `mem::forget` for alloc, and `Vec::from_raw_parts` + `drop` for free.

### Meta buffer layouts

To call a Rust kernel, JS packs metadata into a `Uint32Array`, copies it into WASM memory (`alloc_u32` + write), passes the pointer, and frees afterwards. Layouts (see [src/kernels/utils.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/src/kernels/utils.ts)):

All builders use the shared `MAX_RANK = 64` (imported from `@webtensor/ir`). Offsets into each layout are centralized in Rust (`rust/src/ops/mod.rs`) and mirrored in TS (`src/kernels/utils.ts`) â€” kernels never hardcode offsets.

| Op family         | Words                  | Layout                                                                                                                                     |
| ----------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| Unary             | `3 + 2Â·MAX_RANK` (131) | `[total, rank, offset, shape[MAX_RANK], strides[MAX_RANK]]`                                                                                |
| Binary            | `4 + 3Â·MAX_RANK` (196) | `[total, rank, a_offset, b_offset, out_shape[MAX_RANK], a_bcast_strides[MAX_RANK], b_bcast_strides[MAX_RANK]]`                             |
| Reduce (sum/mean) | `3 + 3Â·MAX_RANK` (195) | `[in_rank, reduce_rank, offset, in_shape[MAX_RANK], in_strides[MAX_RANK], axes[MAX_RANK]]`                                                 |
| Softmax           | `3 + 2Â·MAX_RANK` (131) | `[rank, axis, offset, shape[MAX_RANK], strides[MAX_RANK]]`                                                                                 |
| Concat            | `5 + 3Â·MAX_RANK` (197) | `[num_inputs, axis, total, out_rank, in_rank, out_shape[MAX_RANK], in_strides[MAX_RANK], in_offsets[MAX_RANK]]` (+ per-input meta chunks)  |
| Pad               | `3 + 4Â·MAX_RANK` (259) | `[rank, total, offset, in_shape[MAX_RANK], in_strides[MAX_RANK], pads_before[MAX_RANK], pads_after[MAX_RANK]]`                             |
| MatMul (batched)  | `10 + 3Â·(MAX_RANK - 2)` (196) | `[batch_rank, M, K, N, a_row_stride, a_col_stride, b_row_stride, b_col_stride, a_offset, b_offset, batch_out_shape, a_bcast_batch, b_bcast_batch]` (each batch array sized `MAX_RANK - 2 = 62`) |

Dtype scaffolding: reduction and softmax Rust kernels are named with an `_f32_` suffix (`reduce_sum_f32_strided`, etc.); the TS wrapper looks the function up via `dtypeSuffix(dtype)` so adding `int32` / `bool` variants later is additive, not a rewrite.

The Rust side reads the same layout (e.g. [kernels/activation/relu/forward.rs](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/rust/src/kernels/activation/relu/forward.rs)) using `slice::from_raw_parts(meta_ptr, UNARY_META_SIZE)` where the size constants live in `rust/src/ops/mod.rs`.

### `_raw` pointer convention

Each Rust kernel exposes `<op>_strided(a_ptr, [b_ptr,] out_ptr, meta_ptr)`. ReluBackward has a special `_raw` form (`relu_backward_raw(grad, a, out, n)`) that skips the meta buffer because autograd always passes contiguous tensors â€” see [src/kernels/activation/relu/backward.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/src/kernels/activation/relu/backward.ts).

### Stride utility (Rust)

[rust/src/utils.rs](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-wasm/rust/src/utils.rs) has `strided_idx(shape, strides, offset, flat_idx)` â€” same algorithm as the JS version, in Rust. Used by every kernel that handles arbitrary strides.

### Performance profile

Faster than CPU on large element-wise ops, but with WASM â†” JS boundary overhead. Best for medium-sized workloads.

## WebGPU backend

[packages/backend-webgpu](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/) â€” WGSL compute shaders, async device init.

### Backend class

[backend.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/src/backend.ts):

- `WebGPUBackend.create()` â€” `await navigator.gpu.requestAdapter()` then `requestDevice()`.
- `allocate()` â€” creates a `GPUBuffer` with `STORAGE | COPY_SRC | COPY_DST`.
- `read()` â€” async readback via a staging buffer (`mapAsync(READ)`).
- `write()` â€” `device.queue.writeBuffer(...)`.
- `execute()` â€” pipeline cache lookup, build bind group, dispatch, schedule temp-buffer cleanup.
- `dispose()` â€” `buffer.destroy()` (no-op for views).

### `WebGPUKernel` interface

Each kernel implements three methods (see [utils.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/src/kernels/utils.ts)):

```ts
interface WebGPUKernel {
  createPipeline(device): GPUComputePipeline;
  buildBindGroupEntries(
    device,
    node,
    inputs,
    outputs,
  ): { entries: GPUBindGroupEntry[]; tempBuffers: GPUBuffer[] };
  getDispatch(node, inputs, outputs): [number, number, number];
}
```

`tempBuffers` are meta uniforms allocated per-execute. The backend destroys them after `device.queue.onSubmittedWorkDone()` resolves â€” destroying earlier corrupts the dispatch.

### `TensorMeta` uniform

Defined in [utils.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/src/kernels/utils.ts) â€” **520 bytes, 130 Ă— u32** (rank up to `MAX_RANK = 64`):

```text
[0]        rank
[1]        offset
[2..65]    shape[0..63]   (1-padded)
[66..129]  strides[0..63] (0-padded)
```

WGSL counterpart (emitted by `renderWgsl()` / `injectMeta()` â€” kernels use the `__TENSOR_META__` placeholder):

```wgsl
struct TensorMeta {
  rank:    u32,
  offset:  u32,
  shape:   array<u32, 64>,
  strides: array<u32, 64>,
};
```

Access in WGSL: `u_meta.shape[ax]` â€” flat `array<u32, 64>`, not vec4-packed.

`packMeta(tensor, outShape?)` builds the array. When `outShape` is provided (binary ops), it computes broadcast strides via `broadcastStridesOf()`.

`createMetaBuffer(device, data)` creates a `UNIFORM | COPY_DST` buffer with `mappedAtCreation: true` â€” proven reliable in Chromium.

### The `meta` keyword pitfall

`meta` is a **reserved WGSL keyword**. Always name the uniform `u_meta`, `u_meta_a`, `u_meta_b`. Using bare `meta` causes a silent compile failure: the pipeline builds, but every output is zero. This bit hard once already â€” see [.claude/CLAUDE.md](https://github.com/hongjiaherng/webtensor/blob/main/.claude/CLAUDE.md).

### Bind group convention

| Op family        | Bindings (must be consecutive)                                                                                         |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------- |
| Unary            | 0: input, 1: output, 2: `u_meta`                                                                                       |
| Binary           | 0: a, 1: b, 2: out, 3: `u_meta_a`, 4: `u_meta_b`                                                                       |
| MatMul (batched) | 0: a, 1: b, 2: out, 3: `u_meta_a`, 4: `u_meta_b`, 5: `u_batch` (BatchMeta: batch_rank, M, K, N, batch_out_shape)       |
| Reduce           | 0: in, 1: out, 2: `u_meta_a`, 3: `u_reduce` (kept_rank, reduce_rank, kept_total, reduce_total, kept_axes, reduce_axes) |
| Softmax          | 0: in, 1: out, 2: `u_meta_a`, 3: `u_sm` (axis, slice_count, axis_len, out_strides)                                     |

Auxiliary uniforms (`u_batch`, `u_reduce`, `u_sm`) are created via `createUniformBuffer(device, data)` in `packages/backend-webgpu/src/kernels/utils.ts` â€” arbitrary-length uniforms as opposed to the fixed 520-byte `TensorMeta`.

No gaps allowed â€” WebGPU rejects bind groups with sparse indices.

### Dispatch sizing

Most elementwise/binary kernels use `@workgroup_size(64)` and dispatch `ceil(N / 64)` workgroups. Reductions and Softmax use `@workgroup_size(64)` with one invocation per output element (reductions) or per slice (softmax). Batched MatMul uses `@workgroup_size(8, 8, 1)` and dispatches `[ceil(M/8), ceil(N/8), batchTotal]` so `gid.z` is the flat batch index.

### Pipeline cache

[backend.ts](https://github.com/hongjiaherng/webtensor/blob/main/packages/backend-webgpu/src/backend.ts) caches compiled pipelines by op name. First call to an op compiles the WGSL; subsequent calls are zero-cost lookup.

### Rank-64 ceiling

`TensorMeta` carries 64 shape and 64 stride slots (`MAX_RANK = 64`). Ops allocate with `rank > 64` throw at `allocate()` time on every backend â€” no silent corruption. Batched MatMul caps `batch_rank` at `MAX_RANK - 2 = 62` so the total tensor rank stays within the cap.

### Performance profile

Native GPU parallelism â€” significantly faster on large tensors. Init cost is high (driver wakeup), so amortize across many ops.

## Sync / async surface

| Backend | `execute()` returns                      | Init                                 |
| ------- | ---------------------------------------- | ------------------------------------ |
| CPU     | `void` (synchronous)                     | sync                                 |
| WASM    | `void` (synchronous after init)          | async (no-op in practice)            |
| WebGPU  | `void`, but GPU work is queued and async | async (real wait for adapter/device) |

`engine.evaluate()` `await`s every `execute()` call so the same engine code works for all three.

## Adding a kernel â€” quick map

The full recipe is in [Adding an op](/docs/advanced/adding-an-op). This is the cliff notes:

1. **CPU:** new file under `kernels/<category>/`, register in `kernels/registry.ts`.
2. **WASM:** Rust function with `#[wasm_bindgen]` in `rust/src/kernels/<category>/`, JS wrapper in `src/kernels/<category>/`, register, **rebuild WASM**.
3. **WebGPU:** `<op>.wgsl` (uniform = `u_meta`!) plus `<op>.ts` implementing `WebGPUKernel`, register.
4. **Tests:** add to `tests/ops/<category>/<op>.test.ts` + ensure `tests/backends/consistency.test.ts` covers it.

Continue with [Autograd](/docs/onboarding/autograd).
