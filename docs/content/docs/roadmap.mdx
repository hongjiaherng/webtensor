## Priority: Next Three

### 1. Dtype Kernels (int32 + bool)

**What it is:** Extend op kernels beyond `float32` to support `int32` (and `bool`). The dtype infrastructure is already in place ŌĆö `DType`, `bytesPerElement`, `typedArrayCtor`, dtype-aware `buf()`, and the `dtypeSuffix()` scaffolding in WASM utils. What's missing is per-dtype kernel implementations.

**Why it matters:** `int32` is needed for index ops (Gather, Scatter, embedding lookups ŌĆö required for NLP / classification). `bool` is needed for masks and logical ops.

**What needs to change:**

- **CPU kernels** ŌĆö each existing op branches on `inputs[0].dtype`. Because `buf()` already returns a `TypedArray` union, most kernels just need to substitute an accumulator type and rename `Float32Array` ŌåÆ `typedArrayCtor(dtype)(...)` for temp buffers. Comparison ops (`Eq`, `Lt`, `Gt`, `Ne`) produce `bool` outputs.
- **WASM kernels** ŌĆö add per-dtype Rust functions (`add_i32_strided`, `add_u8_strided`, etc.). The existing `dtypeSuffix(dtype)` helper in `src/kernels/utils.ts` already drives function-name lookup ŌĆö add the matching `_i32_` / `_u8_` entries on `WebtensorWasmModule` and wire them up. Meta layouts are already dtype-independent.
- **WebGPU kernels** ŌĆö WGSL has no generics. Ship one `.wgsl` per dtype per op; TS wrappers select pipeline by `inputs[0].dtype`.
- **Promotion rules** ŌĆö adopt PyTorch rules: `add(int32, float32) ŌåÆ float32`, etc. Define once in a `resultDType(a, b)` helper.

### 2. Loss functions and training modules

**What it is:** A `@webtensor/core/nn` (or similar) module with `mse_loss`, `cross_entropy`, `log_softmax`, plus a `Parameter` convention so weights can be managed uniformly.

**Why it matters:** Today training uses `reduceMean(mul(diff, diff))` ad-hoc. Named losses + `log_softmax` unlock classification training. This also paves the way for an Adam optimizer and richer schedulers.

**What needs to change:**

- **`log_softmax`** in `ops.ts` ŌĆö numerically stable: `x - logsumexp(x, axis)`. Backward: `grad - softmax(x) * sum(grad, axis, keepdim=true)`.
- **`mse_loss(pred, target)` = `reduceMean(pow(sub(pred, target), 2))`** ŌĆö one-liner alias.
- **`cross_entropy(logits, target)`** ŌĆö combined `log_softmax` + negative-log-likelihood.
- **`optim.Adam`** ŌĆö another host-side optimizer next to `SGD` in `packages/core/src/optim.ts`. Keeps the existing placeholder-fed pattern (weights live as `Float32Array`s, not inside the graph).

---

### 3. Concat / Gather / Slice backward

**What it is:** Fill the remaining gaps in the op set that unblock real models: `concat(tensors, axis)`, `gather(data, indices, axis)`, and a backward for `slice` (requires `Pad` or `Scatter`).

**Why it matters:** `concat` is everywhere (skip connections, attention). `gather` with int32 indices enables embedding lookups. `slice` backward is needed for any training code that slices into tensors.

**What needs to change:**

- **`concat`** ŌĆö kernel copies N input tensors into one output along the concat axis. Backward: slice the gradient back into N pieces.
- **`gather`** ŌĆö blocked on int32 kernels (indices must be int32). Kernel reads `data[indices[i]]` per output element. Backward: `scatter_add`.
- **`slice` backward** ŌĆö implement `pad` / `scatter` to write the gradient into a zero tensor at the slice region.
